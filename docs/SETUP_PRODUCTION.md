# OmniChat Production Deployment Guide

This guide explains how to deploy the OmniChat system to a production environment. 

OmniChat is designed with a decoupled architecture:
1. **The Backend (NestJS API & Socket.io):** Handles business logic, database operations, and real-time WebSocket connections.
2. **The Frontend (Vue 3 Web Components):** Compiled into standalone JavaScript files (`omnichat-widget.js` and `omnichat-admin.js`) that can be embedded into *any* website or framework (Blazor, WordPress, React, plain HTML) like a standard script tag.

---

## Phase 1: Deploying the Backend (NestJS API)

The backend needs a Bun environment and a database (e.g., PostgreSQL). It can be hosted on a VPS (DigitalOcean, AWS, Azure), Railway, Render, or within a Docker container.

### 1. Setup and Build
```bash
cd apps/api
bun install
```

### 2. Configure Environment Variables
Create a `.env` file in the `apps/api` directory:
```env
# Database connection string
DATABASE_URL="postgresql://user:password@localhost:5432/omnichat"

# Master signing key for all OmniChat-generated JWTs.
# Must be a strong random string in production.
JWT_SECRET="your-super-secure-random-secret-key"

# Port for the API to listen on
PORT=3001

# Lifetime of the access token JWT returned by login/register/refresh.
# Sent as Authorization: Bearer on every authenticated request.
ACCESS_TOKEN_EXPIRY=30m

# Lifetime of the opaque refresh token returned by login/register/refresh.
# Stored in localStorage, exchanged for new tokens via POST /auth/refresh.
REFRESH_TOKEN_EXPIRY=7d

# Pre-shared HS256 secret used to verify JWTs generated by the embedding site.
# The external site signs a JWT with this key and passes it via the
# data-external-token attribute on the chat widget.
EXTERNAL_SITE_JWT_SECRET="your-external-site-secret-key"

# Admin API Key — required for admin endpoints (register, login, update-password)
ADMIN_API_KEY="your-admin-api-key"
```

### 3. Database Migration
Apply the database schema:
```bash
bun prisma generate
bun prisma migrate deploy # Use 'deploy' for production, not 'db push'
```

### 4. Build and Run
Compile the TypeScript code and start the server:
```bash
bun run build
bun run start:prod
```
*Note: In production, it is highly recommended to run the app using a process manager like **PM2** (`pm2 start dist/main.js --name omnichat-api`) or run it inside a Docker container to ensure it restarts automatically if it crashes.*

### 5. Reverse Proxy & SSL (Crucial)
You must place the API behind a reverse proxy (like Nginx) and secure it with SSL/TLS (HTTPS). WebSockets (`wss://`) require a secure connection to function properly in modern browsers. 
Ensure your Nginx config upgrades the connection for WebSockets:
```nginx
location / {
    proxy_pass http://localhost:3001;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection "upgrade";
    proxy_set_header Host $host;
}
```

## Phase 2: Building and Hosting the Frontend (Web Components)

The frontend consists of Vue components that we compile into raw JavaScript files. These do not require a Bun server to run; they are purely static files.

### 1. Build the Bundles
```bash
cd apps/web
bun install

# Build the chat widget for website visitors
bun run build:chat-widget

# Build the admin dashboard
bun run build:admin-portal

# Build the admin floating widget (for backoffice embed)
bun run build:agent-widget
```

This will output three files in the `apps/web/dist` directory:
- `omnichat-chat-widget.js`
- `omnichat-admin-portal.js`
- `omnichat-agent-widget.js`

### 2. Host the Static Files
You need to host these `.js` files so they are publicly accessible via a URL. You **do not** need to manually copy them into your host application's (like Blazor's) `wwwroot` anymore.

You can host them on:
- **A CDN** (e.g., AWS CloudFront, Cloudflare).
- **Static Hosting** (Vercel, Netlify, GitHub Pages).
- **Your own Nginx Server** (e.g., serving them from `/var/www/omnichat-cdn/`).

Once hosted, you will have URLs like:
- `https://cdn.your-domain.com/js/omnichat-chat-widget.js`
- `https://cdn.your-domain.com/js/omnichat-admin-portal.js`
- `https://cdn.your-domain.com/js/omnichat-agent-widget.js`

## Phase 3: Integrating into Your Host Application

Now that your API is running and your JS files are hosted publicly, you can embed OmniChat into **any** website.

### Integrating the Chat Widget (For Visitors)

**Without external site authentication (public visitors):**
```html
<script src="https://cdn.your-domain.com/js/omnichat-chat-widget.js" defer></script>
<omnichat-chat-widget
    server-url="https://api.your-domain.com"
    bubble-color="#4F46E5"
    welcome-message="Hi there! How can we help you today?"
></omnichat-chat-widget>
```

**With external site authentication (authenticated users):**

The embedding site must generate an HS256-signed JWT on the server side with these claims:
```json
{
  "sub": "<external_user_id>",
  "username": "<display_name>",
  "operatorName": "<site_identifier>",
  "iat": 1717000000,
  "exp": 1717000900
}
```
Signed with the same `EXTERNAL_SITE_JWT_SECRET` value configured in OmniChat's `.env`. The JWT should be short-lived (5–15 minutes recommended). Pass it via the `data-external-token` attribute:
```html
<script src="https://cdn.your-domain.com/js/omnichat-chat-widget.js" defer></script>
<omnichat-chat-widget
    server-url="https://api.your-domain.com"
    data-external-token="eyJhbGciOiJIUzI1NiIs..."
    bubble-color="#4F46E5"
></omnichat-chat-widget>
```

### Integrating the Admin Dashboard
Create an admin page in your application and add this snippet:
```html
<script src="https://cdn.your-domain.com/js/omnichat-admin-portal.js" defer></script>
<omnichat-admin-portal
    server-url="https://api.your-domain.com"
></omnichat-admin-portal>
```

### Integrating the Admin Floating Widget (For Agent Backoffice)
```html
<script src="https://cdn.your-domain.com/js/omnichat-agent-widget.js" defer></script>
<omnichat-agent-widget
    server-url="https://api.your-domain.com"
></omnichat-agent-widget>
```

## Authentication Architecture

### Bearer Mode (default, recommended)

JWTs are stored in `localStorage` and sent via `Authorization: Bearer` header. Each request also includes an `x-external-site-token` header (a companion JWT bound to the login origin).

**Login response:**
```json
{
  "accessToken": "eyJ...",     // JWT, short-lived (30m default), includes boundOrigin
  "refreshToken": "uuid...",    // Opaque token, longer-lived (7d default)
  "siteToken": "eyJ...",        // Origin-bound companion JWT, sent as x-external-site-token
  "user": { "id": "...", "username": "...", "role": "developer" }
}
```

**Refresh flow:** On 401, the client calls `POST /auth/refresh` with the refresh token to get new tokens.

### Roles

| Role | Description |
|------|-------------|
| **developer** | Full access. Can manage CORS, IP allowlist, and promote users to developer. |
| **admin** | Can manage AI agents, knowledge base, tools, logs, and users. Cannot change CORS/IP settings or promote to developer. |
| **agent** | Can view and respond to conversations. Can change widget settings. |

### Why this is better:
- **Decoupled:** Your main website (Blazor, etc.) does not need to store, build, or care about the OmniChat codebase.
- **Auto-Updating:** If you update the OmniChat Vue code, rebuild it, and upload the new `.js` to your CDN, all websites using the `<script>` tag will automatically get the latest version on the next page refresh.
- **Cache Friendly:** Browsers will cache the `.js` file from your CDN, making it load instantly for users navigating between pages.
